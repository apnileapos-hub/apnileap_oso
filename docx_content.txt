1. Introduction
1.1 Purpose
This document is the authoritative design specification for ApniCart, a compact business-to-consumer (B2C) e-commerce storefront commissioned by Infosys and delivered by a student team at KLE Technological University. The engagement is brokered, governed and instrumented by the APNILEAP platform. The document describes both the application to be built and the collaboration machinery that surrounds it: how the work is sponsored, awarded, executed, tracked in Jira, documented in Confluence, version-controlled in GitHub, and accepted.
It is written to serve three audiences simultaneously О“Г‡Г¶ the Infosys sponsor who needs assurance on scope and governance, the KLE faculty and students who need an implementable blueprint, and the APNILEAP PMO that orchestrates the lifecycle. Where a section is primarily relevant to one audience it is flagged accordingly.
1.2 Background: the APNILEAP collaboration model
APNILEAP is a multi-tenant Software-as-a-Service governance and collaboration operating system that restructures how industry and academia work together. It positions itself as a central hub that connects two classes of autonomous spokes О“Г‡Г¶ academic tenants (universities, research centres) and industry tenants (corporations and R&D labs) О“Г‡Г¶ while retaining the master record for project metadata, governance approvals, identities and financial milestones.
Rather than letting collaborations form through informal email and ad-hoc contracts, APNILEAP routes them through a structured marketplace. An industry partner publishes an opportunity; qualified academic teams bid; the platform awards, provisions tooling, tracks milestones against funding, and enforces data isolation between competing tenants. This project is one concrete instance of that model.
Key principle О“Г‡Г¶ APNILEAP is the system of record; Jira is the execution engine; Confluence is the knowledge base; GitHub is the source of truth for code. The platform keeps these three tools synchronised through automated provisioning and event-driven webhooks so that an executive dashboard reflects operational reality without manual status reporting.

2.4 End-to-end engagement flow
Infosys publishes the О“Г‡ВЈB2C E-Commerce StorefrontО“Г‡ВҐ opportunity with budget band, required skills (React, Spring Boot, PostgreSQL) and a submission deadline.
APNILEAP indexes it and notifies eligible academic teams. KLEО“Г‡Г–s PI submits a bid: a proposal document, the proposed student team, and an indicative plan.
Infosys reviews ranked bids, shortlists KLE, negotiates briefly over the collaboration messaging thread, and awards the project.
On award, the provisioning orchestrator creates: a Jira project (key ECOM), a Confluence space (key ECOM) seeded with governance templates, and a GitHub repository from the standard baseline.
The five funded milestones become Jira Epics; the kickoff and charter pages are generated in Confluence; the team begins Sprint 1.
As each milestone Epic reaches Done, a Jira webhook fires a MILESTONE_REVIEW in APNILEAP. The Infosys Liaison verifies the deliverable against acceptance criteria.
On approval, APNILEAP records the gate and triggers the Finance Gateway (RazorpayX) to release the milestone stipend tranche; the student credit ledger is updated.
After the final milestone, APNILEAP closes the project, archives the Confluence space read-only, transfers the GitHub repository per the IP terms, and prompts both parties for a rating.

3. Product Scope: ApniCart
3.1 Problem statement and objectives
Infosys wants a clean, modern reference storefront that a small or mid-sized retailer could adopt: a shopper can discover products, add them to a cart, check out and pay; a store administrator can manage the catalogue, stock and orders. The goal is a deliberately small, production-shaped application that demonstrates sound full-stack engineering rather than feature breadth.
Objectives: (a) ship a deployable MVP that handles the core purchase journey end-to-end; (b) demonstrate secure authentication, real payment integration and an auditable order lifecycle; (c) produce maintainable, well-tested code with CI/CD; and (d) exercise the full APNILEAP governance loop so the model itself is validated.

4. Solution Architecture
4.1 Architecture principles
API-first: every UI capability is backed by a versioned REST endpoint, mirroring the APNILEAP imperative.
Stateless services: the backend holds no per-user state in memory; sessions and carts live in Redis so any instance can serve any request.
Modular monolith for the MVP: a single Spring Boot deployable organised into clear modules, chosen over microservices to keep a student team productive while preserving clean seams for later extraction.
Secure by default: deny-by-default authorization, server-side validation of every mutation, and payment confirmation that never trusts the client.
Twelve-factor configuration: all environment-specific values come from config/secret stores, never the code.
4.2 System context
ApniCart sits between its users and a small set of external services. APNILEAP surrounds the whole engagement as the governance layer but is not a runtime dependency of the application itself.

4.7 Key sequence flows
4.7.1 Checkout and payment
The client posts the cart to POST /api/v1/orders. The API validates stock and prices server-side and creates an order in CREATED.
The API calls Razorpay to create a payment order and returns the Razorpay order id and amount to the client; the order moves to PAYMENT_PENDING.
The client opens the Razorpay checkout widget and the shopper pays. Razorpay returns a payment id and signature to the client.
The client posts these to POST /api/v1/payments/verify. The API recomputes the HMAC-SHA256 signature with the secret key and compares.
On match, the order moves to PAID inside a single transaction that also decrements stock; a confirmation email is queued. On mismatch, the order moves to PAYMENT_FAILED and nothing is fulfilled.
Independently, Razorpay calls POST /api/v1/payments/webhook; the API verifies that signature too, making settlement confirmation idempotent and resilient to a dropped client response.

Representative DDL for the central entities (Flyway migration V1). Audit columns (created_at, updated_at) are present on every table.
CREATE TABLE users (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email         VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  full_name     VARCHAR(160) NOT NULL,
  role          VARCHAR(20)  NOT NULL DEFAULT 'CUSTOMER', -- CUSTOMER | ADMIN
  email_verified BOOLEAN     NOT NULL DEFAULT FALSE,
  created_at    TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE TABLE categories (
  id    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name  VARCHAR(120) NOT NULL,
  slug  VARCHAR(140) UNIQUE NOT NULL
);
CREATE TABLE products (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  category_id UUID REFERENCES categories(id),
  sku         VARCHAR(64) UNIQUE NOT NULL,
  name        VARCHAR(200) NOT NULL,
  description TEXT,
  price       NUMERIC(12,2) NOT NULL CHECK (price >= 0),
  stock_qty   INTEGER NOT NULL DEFAULT 0 CHECK (stock_qty >= 0),
  image_urls  JSONB NOT NULL DEFAULT '[]',
  active      BOOLEAN NOT NULL DEFAULT TRUE,
  search_tsv  tsvector GENERATED ALWAYS AS
              (to_tsvector('english', name || ' ' || coalesce(description,''))) STORED
);
CREATE INDEX idx_products_search ON products USING GIN (search_tsv);
CREATE INDEX idx_products_category ON products (category_id) WHERE active;
CREATE TABLE orders (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id),
  status      VARCHAR(20) NOT NULL DEFAULT 'CREATED',
  total_amount NUMERIC(12,2) NOT NULL,
  ship_address JSONB NOT NULL,
  placed_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE TABLE order_items (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  order_id   UUID NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  product_id UUID NOT NULL REFERENCES products(id),
  unit_price NUMERIC(12,2) NOT NULL,  -- price captured at purchase time
  quantity   INTEGER NOT NULL CHECK (quantity > 0)
);
CREATE TABLE payments (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  order_id      UUID NOT NULL UNIQUE REFERENCES orders(id),
  rzp_order_id  VARCHAR(64) NOT NULL,
  rzp_payment_id VARCHAR(64),
  status        VARCHAR(20) NOT NULL,  -- PENDING | CAPTURED | FAILED
  amount        NUMERIC(12,2) NOT NULL
);
5.3 Indexing and search strategy
Product search uses a generated tsvector column with a GIN index for relevance-ranked full-text queries О“Г‡Г¶ sufficient for the MVP catalogue. Elasticsearch is the documented upgrade path if the catalogue grows.
Category and price filters are served by partial and composite B-tree indexes restricted to active products.
Order-item prices are denormalised onto the line (unit_price) so historical orders are immune to later price changes.
Hot reads (home page, category listings) are cached in Redis with short TTLs and explicit invalidation on admin writes.

8. Security Design
Security mirrors APNILEAPО“Г‡Г–s zero-trust posture, scaled to an MVP. The guiding rule is that the server trusts nothing from the client О“Г‡Г¶ every price, total and payment outcome is recomputed or verified server-side.
8.1 Authentication and authorization
Passwords are hashed with bcrypt (work factor О“Г«Г‘ 12); plaintext is never stored or logged.
A short-lived JWT access token (15 min) carries the user id and role; a rotating refresh token lives in an HttpOnly, Secure, SameSite cookie.
Spring Security enforces method-level authorization: shopper endpoints require ROLE_CUSTOMER, admin endpoints require ROLE_ADMIN, deny-by-default everywhere else.
Object-level checks ensure a shopper can only read their own orders and cart О“Г‡Г¶ the equivalent of APNILEAPО“Г‡Г–s row-level isolation at the application tier.
8.2 Payment security
The amount sent to Razorpay is computed from the server-side cart, never from a client-supplied total.
Order confirmation requires a valid HMAC-SHA256 signature computed with the secret key; a mismatch fails the order.
The webhook endpoint independently verifies its own signature and is idempotent, so a replayed or duplicated callback cannot double-fulfil or double-charge.
Razorpay keys are environment secrets; the secret key never reaches the browser.
8.3 Data protection
TLS everywhere; HSTS on the production domain.
Secrets (DB credentials, JWT signing key, Razorpay keys) come from Azure Key Vault / GitHub OIDC, never from the repository.
Personal data is minimised; an erasure request anonymises the user record while preserving order integrity for accounting.
Audit log records security-relevant events (login, role change, admin mutation, payment outcome).

9.3.2 Integration mechanics
APNILEAP talks to the Confluence Cloud REST API v2 through an egress gateway that injects OAuth tokens, throttles with a token-bucket and circuit-breaks on Atlassian downtime.
Identities are synchronised via SCIM 2.0: when a student is onboarded in APNILEAP, a matching Atlassian account and space permission are provisioned; on closure they are de-provisioned.
Confluence page-update webhooks are HMAC-SHA256 verified and published to a Kafka topic, keeping enterprise search and in-app notifications current.
Governance artefacts (charter, milestone-gate minutes) are generated programmatically from templates so every project has a consistent, audit-ready record.
9.4 Provisioning automation on award
All three tools are created by a single asynchronous workflow when the marketplace transitions the project to AWARDED, so the team starts Sprint 1 against fully wired infrastructure with zero manual setup.
Marketplace emits a ProjectAwardedEvent on Kafka.
The provisioning orchestrator creates the GitHub repo (Terraform), the Jira project and its milestone Epics (REST v3), and the Confluence space with seeded pages (REST v2).
SCIM provisions Atlassian accounts and maps APNILEAP roles to Jira roles, GitHub teams and Confluence permissions.
Webhooks are registered on Jira and Confluence; the orchestrator persists the returned project key, repo URL and space key against the APNILEAP project record.
A kickoff notification with all three links is dispatched to the team.

11. Quality & Testing
11.1 Test strategy
Testing follows the pyramid: a broad base of fast unit tests, a focused band of integration tests around the API and database, and a thin layer of end-to-end tests covering the critical purchase journey. Quality gates in CI block merges that regress coverage or introduce known vulnerabilities.

11.3 Quality gates
CI fails on any failing test, on coverage below the 70% threshold, or on a new high/critical vulnerability.
The Razorpay flow is tested against test-mode keys; a mocked gateway covers failure and signature-mismatch paths.
A pre-release k6 run must keep P95 latency within NFR-01 at the target concurrency.

13. Acceptance & Closure
13.1 Acceptance criteria
All Must-have functional requirements (FR-01..FR-11) demonstrated end-to-end on UAT.
Non-functional targets in Section 3.5 met and evidenced (test reports attached in Confluence).
A successful, signature-verified payment produces a PAID order, decremented stock and a confirmation email.
Admin can manage the catalogue and advance an order through its fulfilment states.
Production deployment succeeds through the gated pipeline.
13.2 IP and handover
Intellectual-property ownership is fixed in the SOW before the project is awarded, with APNILEAP holding the immutable contribution ledger (commits, documents, approvals) as evidence of authorship. On closure, the GitHub repository is transferred to Infosys per those terms, with KLE and the students retaining the agreed academic and portfolio rights.
13.3 Closure checklist
Final milestone (M5) approved by the Liaison; all funds disbursed.
Closure report published in Confluence; space archived read-only.
Repository transferred; access for transient collaborators revoked via SCIM de-provisioning.
Ratings exchanged by both parties to build platform reputation data.
APNILEAP marks the project COMPLETED then CLOSED.
13.4 Change control
Any change to scope, design or plan after baseline is raised as a change request on the Confluence change log, assessed by the PI and Liaison for cost/schedule impact, and approved by the PMO before the affected Jira items are altered.
